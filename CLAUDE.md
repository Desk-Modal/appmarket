## Structure
All paths are relative to this repo root. Works identically on macOS and Windows.
```
<repo-root>/
├── platform/       # DeskModal core — Rust/Tauri FDC3 2.2 desktop agent
├── plugins/        # Independent plugin projects (build & release separately)
│   └── tradesurface/  # Multi-asset charting plugin
├── marketplace/    # App marketplace infrastructure
├── dist/           # CI output — built binaries, signed services, web apps
├── scripts/        # Developer tooling (setup, build, deploy, sign, launch)
└── specs/          # Cross-cutting specs, personas, ADRs
```

## Principles
- **Platform ships as one binary** — `platform/` is a Cargo workspace producing the DeskModal agent
- **Plugins are independent** — each plugin under `plugins/` builds separately, depends on platform SDK via relative path dep
- **Services use ServiceSDK only** — native services import `deskmodal-service-sdk`, never internal platform crates
- **No duplicate code** — one source of truth per concept, single SDK path
- **Incremental builds** — never rebuild everything, only changed crates/packages
- **Cross-platform** — all scripts, paths, and configs work on macOS, Linux, and Git Bash on Windows
- **Zero hardcoded paths** — every generated config and doc expresses paths relative to the workspace root (or via `${CLAUDE_PROJECT_DIR}` / `$ROOT_DIR`). Never reference a developer's `/Users/...`, `/home/...`, or `C:\Users\...` path.
- **SDK evolution follows the compat ladder** at `specs/compat-ladder.yml` — additive within-minor, breaking changes require a deprecation window per the ladder's policy. Enforced by the `sdk-compat` prod-check domain (`specs/prod-check-gates/sdk-compat.sh`) + nightly `.github/workflows/compat-nightly.yml`.

## Developer environment contract

Deterministic across macOS, Linux, and Git Bash on Windows. Everything the developer needs is inside the workspace after `scripts/setup.sh`:

- `tools/` — workspace-vendored binaries: `codebase-memory-mcp`, `mise`, optionally `github-mcp-server`. Git-ignored. Re-run `scripts/setup.sh` (or `--config-only`) to refresh.
- `mise.toml` — pinned Node / pnpm / Python / protoc versions. `mise install` applies them.
- `activate.sh` — `source ./activate.sh` in a shell to put `tools/` on PATH and activate mise-pinned tools for that shell. Never edits `$HOME` or shell rc files.
- `.mcp.json` — workspace MCP registry. Uses `${CLAUDE_PROJECT_DIR}` for every path so it's identical on every developer's machine.
- `.claude/settings.json` — committed hooks, permissions, and env defaults. Team-wide source of truth.
- `.claude/settings.local.json` (in each sub-repo) — machine-local override generated by setup.sh; not committed.
- `.claude/hooks/drift-check.sh` — SessionStart hook. Hashes the committed env contract (setup.sh, hooks, mise.toml, .mcp.json, settings.json) and re-runs `scripts/setup.sh --config-only` when it drifts. Every developer's next Claude session after a `git pull` self-heals.

## Hallucination & context hygiene (MANDATORY)

- **Context discipline** — every session follows `.claude/rules/context-discipline.md`. Read `.session-state/handoff.md` before acting (the SessionStart hook surfaces it). Handoffs are commit-driven: `PostToolUse:Bash(git commit*)` auto-appends a handoff entry per commit, and you flesh it out on the next turn. The old 40-tool-call / 30-minute triggers are retired — commits are the durable checkpoint. The ≥70%-window threshold stays advisory (consider writing a handoff before `/clear`). Never claim "I don't have context" as an excuse — the handoff IS the solution.
- **CBM first** — always use `search_graph` / `trace_path` / `get_code_snippet` before Grep/Read. See `.claude/rules/code-discovery.md`.
- **No invented APIs** — before referencing a Claude Code setting, hook event, or MCP, confirm it exists in official docs. Speculation is a hallucination vector.
- **Cite file:line** — when making a claim about code, include the exact file path and line number you read. If you didn't read it, say so.
- **No local paths in any artefact** — neither committed files nor generated configs may contain an absolute path that starts with `/Users/`, `/home/`, or `C:\Users\`. Use `${CLAUDE_PROJECT_DIR}`, `$ROOT_DIR`, or computed relative paths.
- **Verification hierarchy** — see `.claude/rules/honesty.md`. Never claim success without the strongest available proof.
- **Evidence gates** — banned phrases ("I believe", "should work", "the tests pass" without output) unless immediately followed by a file path, log excerpt, or exit code. See `.claude/rules/context-discipline.md` §2.
- **Commit-message honesty hook** — `.claude/hooks/commit-message-honesty.sh` (installed as `.git/hooks/commit-msg` by `scripts/setup.sh`) rejects commit messages containing the banned phrases from `.claude/rules/honesty.md` unless the same message cites evidence (file:line, `.prod-check/*`, `.session-state/*`, commit SHA, URL, or `exit=N`). Bypass in emergencies only with `DESKMODAL_LAX=1` — every bypass is appended to `.prod-check/lax-bypass.log` (same audit shape as `qg_honor_lax`). Regression tests: `.claude/hooks/tests/commit-message-honesty.test.sh`.
- **Dead-end registry** — the active handoff lists hypotheses already disproved this session and across sessions. Re-trying them is a hallucination defect.

## Dist Output (populated by `scripts/build-dist.sh` — IS the runtime)

`dist/` is a self-contained, portable installation. Copy it anywhere — `cp -r dist/ /opt/deskmodal-prod/`, ship it inside a tarball, drop it on a USB stick. DeskModal discovers everything it needs relative to its own binary location. No installer, no system paths, no env vars.

### Layout
```
dist/                                     # Everything DeskModal needs to run
├── DeskModal{.exe}                       # Agent binary — sits at the root
├── config/
│   ├── desk.toml                         # install_root marker file
│   └── manifests/                        # copied from platform/manifests
├── plugins/
│   ├── deskmodal.feeds/                  # Per-app plugin (one dir per app)
│   │   ├── plugin.toml
│   │   ├── app/                          # Built web assets (index.html + bundles)
│   │   └── icons/
│   ├── deskmodal.chart/ …
│   ├── deskmodal.watchlist/ …
│   ├── deskmodal.price-feed/             # Per-service plugin
│   │   ├── plugin.toml
│   │   ├── publisher.pub                 # Ed25519 verify key
│   │   └── services/
│   │       ├── deskmodal_price_feed.{dll,so,dylib}
│   │       └── deskmodal_price_feed.{dll,so,dylib}.sig
│   ├── deskmodal.paper-trading/ …
│   ├── optiscript/                       # Standalone plugin (not split)
│   │   ├── plugin.toml
│   │   ├── publisher.{key,pub}
│   │   ├── app/
│   │   ├── icons/
│   │   └── services/optiscript_service.{dll,so,dylib} + .sig
│   └── tradesurface/                     # Bundle meta-plugin
│       ├── plugin.toml                   # [bundle] includes = [...]
│       └── workspaces/                   # Workspace templates only
├── data/                                 # Logs + SQLite + storage (portable)
│   ├── logs/
│   └── storage/
├── releases/                             # --sign only: signed .dmpkg per plugin
└── .last_deploy                          # Incremental rebuild marker
```

Library extension is picked at build time from the host OS: `.dll` on Windows, `.dylib` on macOS, `.so` on Linux. The dist layout is **platform-flat** — one OS per dist — not a multi-arch fat tree.

### How the binary resolves this layout at runtime
`install_root()` in `platform/apps/deskmodal-agent/src-tauri/src/main.rs:310` walks up from `std::env::current_exe()` (up to 5 levels) looking for the marker file `config/desk.toml`. The first parent directory that contains the marker is the install root. Everything downstream is relative to it:

- Plugin discovery → `install_root().join("plugins")`, scanned by `deskmodal-plugin::load_plugins_from_dir` (`platform/crates/deskmodal-plugin/src/lib.rs:276`). Every sub-directory containing a `plugin.toml` becomes an installed plugin. Name is irrelevant — the manifest drives everything.
- WebView assets → `deskmodal-plugin://<id>/app/index.html` is served from `install_root()/plugins/<id>/app/index.html` by the Tauri asset protocol handler (`main.rs:523`).
- Config → `install_root()/config/desk.toml` with fallbacks to `./`, `~/.deskmodal/`, `/etc/desk/` (`config.rs:574`). Relative paths inside `desk.toml` resolve against `install_root()` (`config.rs:542`).
- Service signature verification → each service plugin's `publisher.pub` is read from the plugin's own directory (`deskmodal-service-host/src/cdylib_runner.rs:425`) and used to verify the `.sig` for each cdylib.
- Logs + data → `install_root().join("data")` by default, overridable with `DESKMODAL_DATA_DIR`. Move `dist/` and the logs move with it.

The binary MUST sit at the **root** of `dist/`, not under `dist/platform/<os-arch>/`. If you put it one level deeper, `install_root()` still finds the marker (it walks up 5 levels), but every sibling directory duplicates itself and `build-dist.sh` stops working incrementally. Keep it flat.

### How each component lands in dist
- **Agent binary** (`build-dist.sh:109`) — built with `cargo build -p deskmodal-agent`, copied from `platform/target/{profile}/DeskModal{.exe}` straight to `dist/`.
- **TradeSurface web apps** (`build-dist.sh:143`) — built with `pnpm nx run-many --target=build --all`; each `apps/<name>/dist/` goes to `dist/plugins/deskmodal.<name>/app/`.
- **TradeSurface services** (`build-dist.sh:170`) — `cargo build --lib` per service, cdylib copied to `dist/plugins/deskmodal.<svc>/services/`.
- **Standalone plugins** (`build-dist.sh:206`) — any `plugins/<name>/` with both `plugin.toml` and `service/Cargo.toml` (excluding tradesurface). Service built with `cargo build -p <name>-service`, plus `plugin.toml`, `app/`, and `icons/` copied over.
- **Monolithic-bundle split** (`scripts/split-bundle.sh`) — if the build emits a legacy `dist/plugins/deskmodal/` bundle, `split-bundle.sh` reads its `[[apps.entries]]` and `[[services]]` and emits one per-plugin directory each, generates per-plugin manifests, copies icons, and writes a `tradesurface/` bundle meta-plugin containing only the workspace templates.
- **Icons** — either copied by `split-bundle.sh` from the bundle manifest's `icons.src` field, or by `build-dist.sh:276` from `plugins/tradesurface/icons/*.svg` into each matching per-app `icons/` directory.
- **Manifests** — either copied verbatim from source (standalone plugins) or generated by `split-bundle.sh` (split bundles). Every per-plugin manifest is valid on its own.
- **Config** (`build-dist.sh:289`) — `platform/desk.toml` → `dist/config/desk.toml` (this is the `install_root` marker), `platform/manifests/` → `dist/config/manifests/`.
- **Signing** (`build-dist.sh:299`, `--sign` only) — generates a publisher keypair per plugin via `dmpkg keygen` if missing, then signs each service cdylib producing `<lib>.sig` alongside it.
- **Per-plugin releases** (`build-dist.sh:344`, `--sign` only) — runs `dmpkg release <src> --key <publisher.key>` for each standalone plugin, producing signed `.dmpkg` tarballs in `dist/releases/` — the artefact shape the appmarket aggregator consumes.
- **Incremental deploy** — `.last_deploy` timestamp drives `needs_rebuild`/`needs_deploy`; nothing rebuilds or recopies unless source is newer.

### User data location
User data is **NOT** in `~/.deskmodal/` for a portable install — it's under `dist/data/`, so moving `dist/` moves logs/storage/keys with it. The `~/.deskmodal/` path is only used when an installed build is launched without a `config/desk.toml` marker alongside the binary, or when explicitly overridden via `DESKMODAL_DATA_DIR`.

## Cross-Project Dependencies
Services depend on the platform SDK via relative path:
```toml
# In plugins/tradesurface/services/price-feed/Cargo.toml:
deskmodal-service-sdk = { path = "../../../platform/crates/deskmodal-service-sdk" }
```

## Runtime Directory (user machine)

Portable installs keep everything under the `dist/` tree the user unpacked — logs and storage live at `<install-dir>/data/`, not in the user home. The only time DeskModal touches `~/.deskmodal/` is (a) when it was launched without a `config/desk.toml` marker next to the binary, so `install_root()` fell back to home, or (b) when `DESKMODAL_DATA_DIR` is explicitly set.

For a portable install at `<install-dir>/` (any path):
```
<install-dir>/
├── DeskModal{.exe}        # Binary
├── config/desk.toml       # install_root marker (walk-up target)
├── plugins/<id>/...       # One sub-directory per installed plugin
└── data/                  # All user data (logs, storage, keys)
    ├── logs/
    └── storage/
```

For the fallback home-directory mode (non-portable, no marker next to binary):
```
~/.deskmodal/              # Linux/Windows; macOS uses ~/Library/Application Support/deskmodal/
├── desk.toml              # User config
├── plugins/<id>/...       # Installed plugins (same layout as dist/plugins/)
├── logs/
└── data/
```

Both modes use the **same** per-plugin directory shape (`<id>/plugin.toml`, `<id>/app/`, `<id>/services/`, `<id>/publisher.pub`) — the plugin manager doesn't care which root it's scanning.

## Code Discovery (codebase-memory-mcp — MANDATORY)

A knowledge graph MCP indexes all code across platform + plugins. **Use it BEFORE Grep/Glob/Read.**

### Session Bootstrap (every session start)
```
list_projects()   # Discover the names CBM generated for THIS machine —
                  # they're derived from the repo path (e.g.
                  # `Users-alice-deskmodal-platform` vs
                  # `home-bob-projects-deskmodal-platform`). Never hardcode
                  # names in skills, rules, or session prompts.
index_status(project=<platform-project>)
index_status(project=<tradesurface-project>)
manage_adr(project=<platform-project>, mode="get")
```

### When to use (ALWAYS for code questions)
| Need | Tool | Why better than grep |
|------|------|---------------------|
| Find a function/type | `search_graph(project, query)` | Ranked by relevance, file:line |
| Trace a call chain | `trace_path(project, from, to)` | Automatic traversal, not manual |
| Read source by name | `get_code_snippet(project, qualified_name)` | No path lookup needed |
| Understand structure | `get_architecture(project, aspects)` | Instant crate overview |
| Scope a change | `detect_changes(project)` | Maps git diff to impacted symbols |
| Record a decision | `manage_adr(project, mode="update", content)` | Persists across sessions |

### After structural changes
```
index_repository(repo_path="<path>/platform", mode="fast")
```

See `.claude/rules/code-discovery.md` for the full tool reference.

## Verification Standards
- NEVER claim success without proof (logs, test output, screenshots)
- "It works" requires evidence — a passing test or verified output
- When uncertain, state what you know and what you don't
- **Always verify through the canonical pipeline** — every verification
  invocation uses `scripts/local-ci.sh` or `scripts/launch.sh --verify`
  per `.claude/rules/verification.md`. Raw `cargo build|test|check|run`
  and `pnpm nx build|test` are development conveniences, never
  verification. Enforced by `scripts/audit-verify-discipline.sh`,
  wired into `scripts/local-ci.sh --fast` via the `hooks:*` gate
  (`.claude/hooks/tests/audit-verify-discipline.test.sh`). When
  adding a Verification command to a spec, route it through the
  canonical pipeline — don't invoke cargo/pnpm/npm/npx directly.
- **Launch with signed plugins** — GUI / FDC3 / plugin / dist changes
  are only verified by `scripts/launch.sh --verify` (runs `local-ci
  --full --sign`, which includes `build-dist --release --sign`).
  Runtime behaviour assertions live in the Rust + TS test suites
  exercised by `local-ci`, plus Playwright GUI specs under
  `tests/gui/` (invoked via `scripts/prod-check.sh gui`). `target/`
  output is not the runtime; the signed `dist/` is.
